# Правила принятия решений по карьерным направлениям

Этот файл подключается к промптам 02 и 03 как формализованные правила выбора направлений.

## 10 правил выбора направлений (по приоритету)

### Правило 1: Конкуренция важнее числа вакансий

Не "сколько вакансий", а "сколько вакансий НА 100 специалистов".

- 2000 вакансий Backend Java при конкуренции 1.5/100 хуже, чем 500 вакансий DevOps при 10/100
- Приоритизировать направления с ratio >= 6/100
- Не рекомендовать как основной трек направления с ratio < 3/100, если у кандидата нет явного конкурентного преимущества
- Шкала: >= 10 = низкая конкуренция (рынок кандидата), 3-9 = средняя, < 3 = высокая (рынок работодателя)

### Правило 2: Домен важнее стека

Доменная экспертиза - защита от AI и от толпы. Рекомендовать переход внутри домена:

- Энергетика -> IT в энергетике
- Финансы/банки -> финтех
- Ритейл -> product analytics в ecom
- Промышленная автоматизация -> Industrial IoT / MLOps

Домен + стек = уникальное позиционирование, которое невозможно скопировать за курс.

### Правило 3: Мост обязателен при сужающемся стеке

Если текущий стек сужается - не бросать сразу. Использовать как краткосрочный мост:

- Мост = быстрый оффер на текущих навыках, стабилизация дохода
- Параллельно строить следующий шаг
- Пример: C# сужается -> работать как C# тимлид (мост), параллельно осваивать Java/Go

### Правило 4: Non-IT -> IT вход

НИКОГДА не рекомендовать ML/DS/DevOps как первый шаг для non-IT кандидатов.

Путь входа:
1. Системный аналитик / бизнес-аналитик / дата-аналитик
2. Через 1-2 года опыта - рост в Data Engineer, Product Analyst, и далее
3. ML/DS - только при наличии математического бэкграунда и готовности к 12+ мес обучения

Исключение: если есть техническое образование (математика, физика, CS) - DA/BA с ускоренным треком.

### Правило 5: Английский = ворота на рынок

| Уровень English | Доступные рынки | Потолок зарплаты |
|----------------|-----------------|------------------|
| Нулевой (A0-A1) | Только РФ/СНГ | ~300-500к руб/мес |
| Базовый (A2-B1) | РФ/СНГ + подготовка к EU | ~400-600к руб |
| Средний (B1-B2) | EU remote возможен | EUR 4-7к/мес |
| Хороший (B2-C1) | Полный международный | EUR 5-10к/мес |
| Native (C1-C2) | Любой, включая US | $8-15к/мес+ |

Без English B1+ не рекомендовать EU/US remote как основной трек. Указывать English как prerequisite.

### Правило 6: Хайп-фильтр

Не рекомендовать ML/AI как основной трек, если:

- Нет математического бэкграунда (линейная алгебра, статистика, вероятность)
- Кандидат не готов к 12+ мес обучения до первого оффера
- Нет четкого доменного применения (generic "хочу в AI" = слабый сигнал)

Альтернатива: AI-adjacent роли (MLOps, Data Engineering, Backend + AI integrations) - вход проще, спрос высокий.

### Правило 7: Искать "белые пятна"

Направления, где спрос значительно превышает предложение:

- DevOps специализации (MLOps, DevSecOps, FinOps, Platform Engineering) - ниже конкуренция чем "чистый" DevOps
- Industrial IoT / Manufacturing AI - мало кандидатов с промышленным бэкграундом
- Нишевый домен + технический стек = малая конкуренция
- Фильтр: если кандидат может закрыть нишу, где ratio > 10/100 - приоритизировать

### Правило 8: Реалистичные сроки перехода

НЕ ставить абстрактные "6-9 мес" по умолчанию. Сроки зависят от кандидата:

Калибровка: 10+ ч/нед — достаточно для серьёзного перехода. "Критически мало" — только при < 5 ч/нед. Не занижай бюджет времени кандидата.

Факторы, ускоряющие:
- Высокая мотивация + > 10 ч/нед на обучение
- Сильный технический бэкграунд
- Adjacency > 70% (много переносимых навыков)
- Пример реального кейса: переход в DevOps за 4 мес + оффер за 1.5 мес при перерыве 4 года

Факторы, замедляющие:
- 3-5 ч/нед на обучение (удвоить оценку сроков)
- СДВГ, выгорание, семейные обязательства
- Adjacency < 50% (фактически новая карьера)
- Отсутствие английского при целевом рынке EU

Диапазон с обоснованием. Эксперт (Алиса) корректирует при ревью.

### Правило 9: Зарплата не ниже текущей

Ни одно из 3 направлений не должно предполагать зарплату ниже текущей кандидата.

- Если переход требует временного снижения - явно предупредить
- Показать горизонт возврата к текущему уровню и роста выше
- Пример: "На старте возможен откат до X, но через 6-12 мес выход на Y, что выше текущей Z"
- Если все 3 направления неизбежно ниже текущей зп - это красный флаг, обсудить с экспертом

### Правило 10: Близость навыков определяет вход

Рекомендовать направления, где 50-70%+ навыков уже есть.

- Не рекомендовать переход с 0 в полностью новую область, даже если спрос высокий
- "У тебя уже есть X, Y, Z - нужно подтянуть только A и B" - убедительный аргумент
- Adjacency < 40% - только как долгосрочная ставка, не основной трек
- Чем выше adjacency - тем быстрее оффер и выше вероятность успеха

## Правило суперсилы: не завышать роль кандидата

Суперсила должна отражать ТЕКУЩУЮ роль и реальный опыт кандидата, а не роль мечты или следующий карьерный уровень.

- Если кандидат - аналитик, суперсила формулируется как "аналитик с ...", НЕ как "архитектор"
- Если кандидат - middle-разработчик, не называй его "Tech Lead" или "Architect" в суперсиле
- Рост в следующую роль (SA -> Architect, Senior -> Lead) - это НАПРАВЛЕНИЕ, а не суперсила
- Суперсила = позиционирование для рынка СЕЙЧАС, не через 2 года

Примеры ПРАВИЛЬНЫХ суперсил:
- Аналитик с dev-бэкграундом: "Системный аналитик с глубоким техническим бэкграундом в Java и интеграциях enterprise-систем" (отражает ТЕКУЩУЮ роль)
- Middle-разработчик: "Backend-разработчик с доменной экспертизой в финтехе" (не "архитектор", не "Tech Lead")

Примеры НЕПРАВИЛЬНЫХ суперсил:
- "Системный архитектор" - если кандидат аналитик, а не архитектор
- "Tech Lead высоконагруженных систем" - если кандидат Senior-разработчик без лидерского опыта
- "ML Engineer" - если кандидат Data Analyst без ML-практики

## Правило: нишевые роли (Architect, Staff, Principal) - не основной трек

Роли типа Solution Architect, Enterprise Architect, Staff Engineer, Principal Engineer:
- Очень мало вакансий (нишевый рынок)
- Рекомендовать ТОЛЬКО как долгосрочную цель / горизонт роста
- НЕ рекомендовать как самостоятельное направление для перехода
- Указывать как "куда расти дальше" внутри направления

## Калибровка "Близость перехода, %"

Везде используем русский термин "Близость перехода (%)" вместо "Adjacency Score".

| Диапазон | Значение | Примеры из реальных кейсов |
|----------|----------|---------------------------|
| 95-100% | Та же роль, другой работодатель | Ольга frontend->frontend, Борис DE->DE, Владислава Java->Java |
| 80-95% | Тот же стек, другая специализация | Чынгыз Java fintech СНГ -> Java fintech EU, Кристина BI -> Sr. BI |
| 60-79% | Смежная роль, частичный перенос | Ольга frontend->fullstack 70%, Djanibek PLC->MLOps 75%, Татьяна SA->BA 60% |
| 40-59% | Серьезный переход | Татьяна SA->DevOps 40%, Лариса fullstack->DevOps 55%, Anna Tarnakina нотариус->SA 55% |
| < 40% | Фактически новая карьера | Anna Tarnakina->DevOps 30%, Non-IT->ML/DS |

## Правила по зарплатам

- Если желаемая зарплата через 3-5 лет превышает медианный Senior в регионе более чем в 1.5x - явно указать
- Показать: какой уровень (Senior / Staff / Principal) и какой рынок реально соответствует этой цифре
- Не занижать потенциал - но и не обещать нереалистичного
- При переходе между рынками (РФ -> EU): учитывать, что номинально зарплата может быть выше, но стоимость жизни тоже
