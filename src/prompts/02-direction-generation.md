## Как формулировать направления

Роль + домен + стек:
- ✓ "Backend engineer in fintech, Node.js"
- ✓ "Backend engineer in fintech, Go"
- ✗ "Backend engineer Node.js/Go" — разные стеки = разные направления, не объединяй
- ✗ "Ruby developer" — слишком коротко, нужен домен

**Правило разделения по стеку:**
Если у кандидата несколько языков/фреймворков — каждый стек = отдельное направление.
Также добавляй стеки, выгодные по рынку (ниже конкуренция, выше зп), даже если кандидат на них не работает сейчас.
Пример: фронтендер знает React и Angular → предложи отдельно "Frontend React", "Frontend Vue" (ниже конкуренция), "Fullstack Node.js".

### Обязательно включи (даже если считаешь неоптимальным — отсев на следующем шаге):
- Текущую роль кандидата — **только если это наёмная IT-должность** (не бизнес, не фриланс, не блогинг, не преподавание)
- Все направления из `directionInterest`

### Разнообразие направлений (ВАЖНО):
- **Минимум 3 разные роли** из KNOWN_ROLES (например: frontend, fullstack, devops — а не frontend-react, frontend-vue, frontend-angular)
- Вариации стека внутри одной роли (React vs Vue) — допустимы, но НЕ заменяют разнообразие ролей
- Если кандидат — фронтендер, обязательно предложи fullstack/backend/devops как отдельные направления

### Фильтры:
- **Зарплата**: если медиана направления ниже текущей зп кандидата — не включай (или обоснуй горизонт роста в whyFits).
- **Рисковые направления**: если live вакансий < 200 или конкуренция высокая (ratio < 3/100) — отмечай в whyFits как рисковое. Не ставь такие как основной трек.

### Обоснование цифрами:
В whyFits для каждого направления **обязательно** укажи конкретные числа из рыночной сводки: число вакансий, медиану зп, тренд. Пример: "1,519 live вакансий UK, медиана £70k, конкуренция ниже чем React (636 вакансий)".

### Среди 5-9 обязательно:
1. **Основной трек**: максимальная близость перехода (>=70%), быстрейший путь к офферу с повышением зп
2. **Запасной вариант**: другой профиль риска, расширение опций
3. **Долгосрочная ставка** или **краткосрочный мост**

### Тип стратегии зависит от стека:
- Устойчив, рынок широкий → основной трек = рост (Senior → Lead/Manager)
- Сужается → мост = текущая роль для дохода + параллельный переход
- Мёртв / кандидат хочет уйти → фокус на близость перехода

## Ограничения

- **Frontend → уход из чистого фронта**: чистый фронтенд под давлением AI-замещения. Любая frontend-роль (включая Team Lead frontend) = только мост или запасной, НИКОГДА долгосрочная ставка или основной трек. Основной трек = fullstack/backend/devops/platform/product. Смотри кейсы (#7, #17, #21, #23).
- **Высокий AI-риск = не долгосрочная**: роли с высоким AI-риском (frontend, SA, QA, UX/UI дизайн, техпис) — только мост или запасной. Для долгосрочных ставок выбирай роли с низким AI-риском.
- **Non-IT → IT**: никогда ML/DS/DevOps как первый шаг. Вход через SA / BA / DA. Исключение: тех. образование или научный бэкграунд.
- **Не рекомендуй** фриланс, EdTech, контент, консалтинг как основные треки (если кандидат не просил).
- **Junior → Middle**: оценивай рынок на уровне Middle (вакансий Junior < 5-10% рынка).
- **Хайп-фильтр**: не переоценивай направления с высоким порогом входа (12+ мес) или перегретым рынком.

### Близость перехода (%)

| Диапазон | Значение |
|----------|----------|
| 95-100% | Та же роль, другой работодатель |
| 80-95% | Тот же стек, другая специализация |
| 60-79% | Смежная роль, частичный перенос |
| 40-59% | Серьёзный переход |
| < 40% | Фактически новая карьера (не включай без веской причины) |
