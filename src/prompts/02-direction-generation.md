# Промпт 2: Генерация 10-14 карьерных направлений (shortlist)

## Роль
Действуй как лучший в мире карьерный стратег по 
трудоустройству в IT в 2026 году.

## Задача
Выбери **10-14 лучших IT-профессий** для кандидата на основе рыночных данных и профиля.
«Лучшее» = максимальный рост зарплаты при низкой конкуренции, высоком спросе и минимальном переобучении исходя из анкеты клиента и данных рынка.

Для каждого направления поставь свой интегральный **`score` (0-100)** - «насколько это
направление подходит клиенту исходя из реалий рынка и анкеты клиента». Весь массив `directions` верни **отсортированным по score DESC**
(первым - с самым высоким). Первые ~10 покажем консультанту сразу, остальные пойдут в запас для
быстрой замены, если консультант попросит заменить конкретное направление.

**Важно про уровень обобщения:** это ПРЕДВАРИТЕЛЬНЫЙ shortlist, не финальный анализ. На этом шаге
предлагай роли на уровне профессии, без узких под-ниш/доменов/специализаций. Более узкую нишу
(«Process Automation», «HR Tech», «Enterprise SaaS» и т.п.) определим на следующем шаге глубокого
анализа. Здесь же roleTitle = просто "Business Analyst (senior)", а не "Business Analyst, Process
Automation (senior)".

## Отранжированный scorer'ом top-20 (база для твоего ранжирования)

Детерминированный scorer выдает по каждой роли **score 0-100** как взвешенное среднее 6 компонент (вес в %; компонент скипается если данных нет - вес размазывается):

| Компонент | Вес | Шкала |
|-----------|-----|-------|
| **salary** (s) | 30 | roleSalary@grade vs desiredSalary: ≥95% → 100 · 85-95% → 80 · 70-85% → 55 · 50-70% → 30 · <50% → 10 |
| **adjacency** (adj) | 22 | slug=current → 100 · slug∈desired → 95 · same cat + **same stackFamily** → 100 · same cat + разный стек → 70 · иначе CATEGORY_BRIDGE (fullstack-JS ↔ backend_nodejs = 100, fullstack-JS → backend_go = 70, dev → devops = 55, non-IT → 40) |
| **market** (m)     | 22 | log-scale от числа вакансий (1500 ≈ 48, 500 ≈ 40, 150 ≈ 32, 50 ≈ 25) |
| **aiRisk** (ai)    | 15 | low → 100 · medium → 75 · high → 35 · extreme → 5 (и hard-filter) |
| **competition** (c)| 10 | vacancies/100 specialists: ≥10 → 100 · 5-10 → 70 · 3-5 → 45 · 1-3 → 25 · <1 → 5 |
| **trend** (t)      | 5  | now/2yrs: ≥1.2 → 85 · ≥1.0 → 70 · ≥0.85 → 60 · ≥0.6 → 45 · ≥0.4 → 30 · <0.4 → 20 |

Зп считается **на грейде клиента** (senior клиент → senior-медиана, не общий middle+).
Hard-filter (до scoring): vac < 100, aiRisk=extreme, `roleSalary < max(desired×0.85, current×1.10)`
- такие роли отсеиваются, **кроме** текущей/desired клиента (они всегда проходят с пометкой ★).

{{scorerTop20}}

**Как пользоваться scorer-score:**
- Для ролей из scorer-таблицы твой `score` держи **в пределах ±8 от scorer-score** той же роли.
- Scorer уже учел: рынок, конкуренцию, зп на грейде, AI-риск, близость перехода **с учетом
  stackFamily**, тренд. Не пересчитывай это заново.
- Расхождение со scorer должно приходить из того, чего **нет в карточке**: характер клиента,
  языковой барьер, декрет, конкретные «не хочу X» из анкеты, невидимая scorer'у стек-несовместимость.
  Если меняешь порядок - коротко поясни в `whyFits` («у клиента нет опыта Go → backend_go как
  долгая ставка, не как быстрый вход»).
- Расходишься больше чем на 8 - сам себе задай вопрос «у меня реально есть основание, или вкусовщина».

**Правило состава shortlist'а:**
- Минимум **8 из 10-14** направлений - из KNOWN_ROLES (scorer-таблица + каталог).
- Максимум **2 off-index** роли в top-10 (`offIndex=true` + `marketEvidence` обязательно).
- Off-index `score` ставишь по своему пониманию (штраф не накладываем), но помни: без
  market-data у нас меньше уверенности, так что off-index должна быть действительно сильнее
  on-index альтернативы, чтобы попасть в top-5.

**В каждом направлении поле `roleSlug` обязательно:**
- если берешь из KNOWN_ROLES - строго snake_case slug из списка (например, `backend_python`, `devops`)
- если предлагаешь редкую off-index роль - `offIndex: true` + `marketEvidence` (где и сколько вакансий,
  медиана зп, источник в 1-2 предложениях)

**ЖЕСТКИЕ ПРАВИЛА roleSlug (нарушения отбраковываются post-валидатором):**

1. **Уникальность**: большинство `roleSlug` встречаются в массиве directions **ровно ОДИН раз**.
   Если у тебя два кандидата на один slug - оставь ОДНО направление, а вариации опиши **внутри**
   `whyFits` или `transferableSkills`. Примеры склейки:
   - Backend Go + Backend Go в fintech → один `backend_go` (fintech на 1A вообще не упоминаем).
   - React senior + React middle - один `frontend_react` (грейд в title).

   **Исключение - «широкие» семейства** (`infosecspec`, `devops`, `data_engineer`, `ml_engineer`,
   `marketing_manager`, `product_manager`, `ui_ux_designer`, `fullstack`): для них допускается
   до **3 direction'ов с одним slug**, если они покрывают **реально разные ниши** внутри семейства,
   которые предлагают разные карьерные треки для senior/lead-клиента. Пример для senior infosec:
   - `infosecspec` - Security Engineer, AppSec
   - `infosecspec` - DevSecOps Engineer
   - `infosecspec` - Cloud Security Engineer, AWS

   Если слияние в одно направление теряет смысл (разные скиллы, разные работодатели, разный уровень
   зп) - оставь несколько directions с общим slug, но **разными title и whyFits**.

2. **Когда МАППИТЬ на KNOWN_ROLES, а когда ставить `offIndex=true`**:
   - **Маппим** на существующий slug, если роль = вариант специализации / ветвь внутри известной:
     - MLOps, Platform Engineer, SRE, Cloud Engineer → `devops`
     - Growth Manager, Performance Marketer, Product Marketing Manager → `marketing_manager`
     - Solution Architect / Solutions Engineer (архитектурная сторона) → `software_architect`
     - Analytics Engineer (инженер-аналитик в data-стеке) → `data_analyst` или `data_engineer` по уклону
     - **Engineering Manager, Engineering Team Lead, Head of Engineering, CTO-направления → `tech_lead`**
       (в нашем каталоге tech_lead покрывает и техлид, и EM, и управление инженерной командой).
   - **Off-index `offIndex=true` + `marketEvidence` обязательно**, если нет явного родителя:
     - Customer Success Manager - НЕ marketing, НЕ sales в нашем каталоге → `customer_success_manager`
       (off-index)
     - Data Steward, AI Automation Engineer, Developer Advocate, Prompt Engineer - отдельные роли
   - **ЗАПРЕЩЕНО** притягивать роль к первому попавшемуся похожему slug только ради попадания в каталог
     (Customer Success → `marketing_manager` - это ошибка, у них разный рынок и зп).

**Следствие про уникальность + EM/tech_lead:**
Если у тебя уже есть направление с `tech_lead` (например основное направление клиента - техлид),
и ты хочешь отдельно упомянуть Engineering Manager / CTO-путь как альтернативу - это **НЕ второе
направление с другим slug** (project_manager для EM - ошибка). Это упоминание внутри `whyFits`
того же tech_lead направления: «…возможен разворот в сторону Engineering Manager / Head of
Engineering с ~15% премии к зп…». Аналогично для DevOps → Platform/SRE/MLOps внутри одной позиции.

3. **Запрет ролей с aiRisk = extreme**. Такие роли (`technical_writer`, `manual_testing` и т.п.) -
   тупики: из них некуда расти, AI быстро вытесняет. **Не предлагай их вообще.**
   Исключение: если это **текущая** роль клиента и он явно НЕ хочет уходить - одна позиция как
   база для оценки рынка (с низким score - честная картина «что будет, если остаться»).

**Поле `bucket` обязательно**, строго одно из: `"ru"` (RUB, RU-данные) / `"abroad"` (EUR, UK-данные как proxy для EU/UK/LATAM/APAC/ME) / `"usa"` (EUR × 1.5). Конкретные регионы (eu/uk/finland/...) - не в `bucket`, а в `whyFits`.

Правило: **если у клиента есть доступ к загранице - фокус ВСЕГДА на загранице**. `bucket="ru"` ставим только если:
1. англ < B1 (все directions = `"ru"`, в `whyFits` упомяни английский как барьер №1);
2. `targetMarketRegions` = только `ru`/`cis`;
3. `accessibleMarkets` = только `ru`/`cis`;
4. роль локальная (`1c_developer`, госсектор РФ и т.п.) - независимо от таргета.

`"usa"` vs `"abroad"`: `"usa"` - если `us` в `targetMarketRegions` или `physicalCountry = United States`. Без визы (`us` не в `accessibleMarkets`) - помечай `"usa"` только для запасных/длинных вариантов с пометкой про H1B/O-1. Для быстрого входа в таком кейсе - `"abroad"`. Если у клиента и EU, и US в таргете: US только где рынок жирнее и виза есть, иначе `"abroad"`.

Если направление реально релевантно и в РФ, и за границей - оформляй **двумя directions** с разным bucket (общий slug для «широких» семейств разрешен). Если `bucket ∈ {"abroad","usa"}`, но `ru` в таргете - можешь добавить одну короткую сноску в `whyFits` про RU-медиану (справочно), не делая параллельный direction.

## Как формулировать направления (title)

Title = **роль + стек + уровень**, БЕЗ домена/ниши/функциональной специализации. Любую
доменную (fintech, SaaS, EdTech, biotech, e-commerce, gov-tech) или функциональную узкую
специализацию (Process Automation, HR Tech, Growth, Enterprise, Customer-facing, AppSec,
MLOps) **определяем на следующем шаге** глубокого анализа - на Phase 1 ее указывать **запрещено**.

- ✓ "Backend Developer, Go (senior)"
- ✓ "DevOps Engineer (middle+)"
- ✓ "Data Analyst, SQL/Python (middle)"
- ✓ "Business Analyst (senior)"
- ✓ "Product Manager (middle+)"
- ✗ "Backend Developer for fintech, Go" - fintech не нужен
- ✗ "Marketing Manager в SaaS/EdTech" - ниша лишняя
- ✗ "Business Analyst, Process Automation (senior)" - Process Automation = под-ниша, убираем
- ✗ "Product Manager, HR Tech (middle+)" - HR Tech = домен, убираем
- ✗ "Data Analyst, Growth/Marketing (middle)" - Growth = функция, убираем
- ✗ "Backend engineer Node.js/Go" - разные стеки = разные направления

Исключение: стек/язык (Go, Python, React, SQL/Python) - это НЕ ниша, его указывать нужно
(он определяет рынок и зп). Под-нишу (что делать внутри роли) - не указывай.

**Правило разделения по стеку:**
Если у кандидата несколько языков/фреймворков - каждый стек = отдельное направление (и отдельный slug).
Также добавляй стеки, выгодные по рынку (ниже конкуренция, выше зп), даже если кандидат на них не
работает сейчас. Пример: фронтендер знает React → предложи `frontend_react`, `fullstack`, `devops`
как отдельные направления - порядок определит score.

### Уровни (ЖЕСТКО):

Junior/entry-level по умолчанию **запрещены**. Предлагай только `middle / middle+ / senior / lead`.

**Исключение**: если клиент в профиле/анкете явно просит junior/стажировку/entry - тогда можно
и нужно (например, "intern", "стажировка", "готова с нуля, junior позиция" - буквально написано).

**Как выбирать уровень для направления:**

1. **Извне IT → IT** (клиент non-IT, `currentProfessionSlug=null`): входим сразу в **middle**,
   не в junior. Даже если это новая для клиента область - у него есть переносимый опыт (проекты,
   люди, домен) на middle-уровень. Пример: PhD-исследователь → "Data Analyst, SQL/Python (middle)".
   **Senior/lead для non-IT запрещен**, кроме явных lead-титулов в анкете (Art/Creative Director,
   Head of, CEO, CTO, VP of, Director, Principal, Engineering Manager, Founder IT-продукта) -
   они дают `lead`. Никакого автоматического `senior` «потому что 15 лет опыта» для non-IT.

2. **Внутри IT, роль совпадает с текущей** (текущий slug из KNOWN_ROLES и предлагаем тот же): берем
   **фактический грейд из профиля** ("у клиента senior" → "Backend Go (senior)").

3. **Внутри IT, меняет роль внутри одного блока** - грейд **сохраняется**. Блоки IT:
   - **Development**: backend, frontend, fullstack, mobile, embedded (переходы внутри: Ruby → Go,
     frontend → fullstack)
   - **Platform/Infra**: devops, platform, sre, cloud_engineer, security
   - **Data/ML**: data_analyst, data_engineer, data_scientist, ml_engineer, analytics_engineer
   - **Analysis**: business_analyst, system_analyst (в том числе → product/data аналитика)
   - **Product/Management**: product_manager, project_manager, engineering_manager (переходы PM →
     PdM → EM считаем внутри блока)
   - **QA**: manual_testing, automation_qa
   - **Design**: ux_designer, ui_designer

   Примеры: Senior System Analyst → Senior Product Manager (product/mgmt блок с аналитикой частично
   пересекается, но senior сохраняется); Senior Backend → Senior Fullstack; Senior Project Manager
   → Senior Product Manager.

4. **Внутри IT, но переход ВНЕ блока** (напр. backend dev → devops, frontend → product, data analyst
   → backend): уровень **сбивается на один вниз**, минимум `middle+`. Senior dev с 20 годами →
   "DevOps (senior)" - валидно, потому что system-мышление есть; но если опыта в новом блоке меньше -
   "DevOps (middle+)". Junior в новом блоке никогда.

   **Исключение - бизнес/предпринимательский опыт "дожимает" уровень обратно до senior**: если
   у клиента есть релевантный опыт запуска своего продукта/бизнеса/CEO-роль в близкой области -
   целевой грейд восстанавливается до senior (не middle+). Примеры:
   - 20 лет dev + свой IT-бизнес → "Product Manager (senior)" (бизнес дает продуктовое и
     стратегическое мышление поверх тех. базы);
   - senior маркетолог + свой стартап → "Product Manager (senior)";
   - CTO стартапа → "Engineering Manager (senior)".

5. **Уровень ВСЕГДА указывай в title** в скобках: "(middle)", "(middle+)", "(senior)", "(lead)".

### Обязательно включи в shortlist (score объективный - может быть и низкий):

Эти 2 группы **всегда** должны присутствовать в `directions`, даже если их score низкий.
Клиент ждет чтобы мы его услышали (желаемое) и дали ориентир «что будет если остаться» (текущая).

1. **Текущую роль кандидата или ближайший IT-аналог**:
   - Клиент уже в IT и `currentProfessionSlug` из KNOWN_ROLES и aiRisk != extreme -
     возьми этот slug. Низкий score из-за падающего рынка/AI-риска? Нормально: покажет клиенту
     честную картину «что будет если остаться».
   - Клиент не в IT (`currentProfessionSlug=null`) - **обязательно** возьми все slug-и из
     `closestItSlugs` (Phase 0 уже выбрал ближайший функциональный IT-аналог: HR не-IT →
     `recruiter`, менеджер не-IT → `project_manager`/`product_manager`, маркетолог не-IT →
     `marketing_manager` и т.д.). Если `closestItSlugs` пуст - у клиента нет очевидного
     IT-аналога (юрист/врач/инженер вышек), просто пропусти этот пункт.
     Score - объективный: если зп по этому аналогу ниже `desiredSalary` или рынок слабый -
     score может быть средним/низким, но **само направление в списке есть**. Это самый
     короткий путь в IT - его нельзя пропустить.
   - Если ни `currentProfessionSlug`, ни `closestItSlugs` ничего не дают, и в анкете есть
     доменный опыт с очевидным IT-эквивалентом, не покрытым Phase 0 - добавь его сама
     по разделу `Non-IT с доменным опытом → смежная IT-роль` ниже.
   - Исключения когда текущую роль НЕ включаем: aiRisk=extreme (тупик), бизнес/фриланс/блогинг/
     преподавание как «текущая роль» без IT-контекста.

2. **Все направления из `directionInterest` / `desiredDirectionSlugs`** - то, что клиент
   явно просит. Каждое в списке. Если не подходит объективно - `recommended:false` +
   `rejectionReason` + низкий score (0-45). Если подходит - нормальный score.

### Ограничения из профиля
Если в анкете явно сказано «не хочу менеджмент /
frontend / X» - НЕ предлагай эти роли, даже если scorer подсветил.

### Поле `recommended` и `rejectionReason` (ВАЖНО):

Если клиент **сам** просит направление (из `desiredDirectionSlugs`/`directionInterest`), которое
объективно ему не подходит - **все равно включи его** в список, но с `recommended: false` и
заполненным `rejectionReason`. Это честнее для клиента: мы его слышим, но сразу предупреждаем.

Когда ставить `recommended: false`:
- Клиент без технического бэкграунда просит DevOps / ML Engineer / Data Engineer / tech_lead
  (без опыта разработки/devops/data/sysadmin/linux-терминала).
- Клиент-дизайнер / маркетолог / не-IT просит tech_lead (путь туда 3+ года через dev/devops).
- Клиент senior/lead с адекватным доходом просит middle-роль без явной просадки-мост (просадка
  10%+ без цели).
- adjacencyScorePercent < 30% на фоне желания клиента.
- Роль с aiRisk=extreme.

Для `recommended: false` направлений `score` всегда **низкий (0-45)** - даже если рынок и зп роли
объективно хорошие, но клиенту она не подходит, это должно быть видно в score.

Если клиент НЕ просил - не включай объективно неподходящие. `recommended: false` - только для
запрошенных клиентом. Обязательно:
- `rejectionReason` - 1-2 фразы по-человечески, без уничижительного тона: «Для DevOps обычно нужна база Linux/терминал/CI-CD, которой сейчас нет. Если есть
  желание учить - путь ~10-12 мес, но это длинная история.»



### Как писать `whyFits` (для консультанта, не для клиента):
2-4 предложения, ~60 слов. Структура:
1. **Опорный опыт**: конкретика из резюме/анкеты (стек, проекты, домен, годы, цифры из опыта). Не «глубокое понимание X».
2. **Связка**: почему именно этот slug - логичный шаг (или мост / запасной / долгая ставка).
3. *(опц.)* **Сравнение/сигнал**: чем отличается от соседней роли; вход без переобучения / нужен мост 6-12 мес / рынок узкий.

Запреты:
- Не дублировать рыночные цифры (median / vac / тренд / конкуренция): они в карточке. Упомянуть число можно, только если это ключевой аргумент («рынок втрое уже соседнего slug - не как основной»).
- Без канцелярита («глубокое понимание», «системное мышление» и т.п.) и без выдумок опыта, которого нет в анкете/резюме.
- Без оборванных фраз на «…» - режь по смыслу, а не по длине.

Для `recommended:false` `whyFits` нейтрален (что из опыта здесь работает), а **почему не подходит** - отдельно в `rejectionReason`, не дублируй.

## Как считать `score` (0-100)

- **Для ролей из scorer-таблицы**: начинай со scorer-score и держись ±8 (веса и шкалы компонент -
  в разделе «Отранжированный scorer'ом top-20» выше). Расхождение приходит из того, чего scorer
  не видит (характер, рынок, анкета).
- **Для off-index ролей** (нет в scorer): оценивай сам по тем же компонентам - salary (главный,
  на грейде клиента), market, competition, aiRisk, adjacency (с учетом stackFamily), trend.
  Плюс **реалистичность пути**: non-technical → DevOps/ML/tech_lead = очень низкий реализм
  (см. `recommended:false`).

Цветовая калибровка (по `score`):
- **80-100** (🟢) - отлично подходит: зп попадает, рынок живой, AI-риск низкий, близкий стек/категория.
- **55-79** (🟡) - с оговорками: просадка зп <15%, либо высокий AI-риск, либо узкий рынок, либо
  нужен мост.
- **0-54** (🔴) - слабо подходит или `recommended:false`.

Верни массив `directions`, **отсортированный по `score` DESC**.

## Ограничения

- **`tech_lead` - ТОЛЬКО для технического бэкграунда**: slug `tech_lead` в нашем каталоге покрывает
  Tech Lead / Engineering Manager / Head of Engineering / CTO - это управление **инженерной** командой.
  Путь туда идет через разработку (backend/frontend/fullstack/mobile), DevOps/Platform/SRE, Data
  Engineering или Security Engineering. **НЕ предлагай tech_lead** клиентам из:
  - UX/UI дизайна (Art Director, Head of Design) - их путь через PM / Design Lead, не tech_lead
  - маркетинга / PM / BA - их путь через product_manager / project_manager
  - аналитики без кодинга (не data_engineer) - через product_manager / data_engineer
  - не-IT бэкграунда - через middle-роль, не сразу в техлид
  Исключение: если клиент **сам** указал tech_lead в `desiredDirectionSlugs` и у него есть 5+ лет
  разработки/devops/data engineering в резюме.
- **Frontend → уход из чистого фронта**: чистый фронтенд под давлением AI-замещения. Любая frontend-роль
  (включая Team Lead frontend) должна получать **средний score** (не топ), приоритет отдаем смежным
  ролям - fullstack/backend/devops/platform/product. Смотри кейсы (#7, #17, #21, #23).
- **Высокий AI-риск = штраф в score**: роли с высоким AI-риском (frontend, SA, QA, UX/UI дизайн) -
  получают меньший score. Чем дальше горизонт - тем сильнее штраф.
- **AI-риск extreme = НЕ ПРЕДЛАГАЕМ ВООБЩЕ** (см. правило выше про roleSlug).
- **Non-IT → IT**: никогда ML/DS/DevOps как первый шаг. Вход через SA / BA / DA. Исключение: тех.
  образование или научный бэкграунд.
- **Non-IT с доменным опытом → смежная IT-роль (ВАЖНО)**: если у клиента 5+ лет в non-IT
  области, для которой есть прямой IT-аналог - **обязательно включи этот аналог** в shortlist
  (score объективный: если зп/рынок под клиента слабые - может быть и средний score, это ок).
  Это самый короткий путь в IT (адаптация рынка, не переучивание).
  - HR / recruiting / talent (5+ лет) → `recruiter` (IT Recruiter / HR BP в tech-компании)
  - Finance / accounting (5+ лет) → `finops` / `data_analyst` (финансовая аналитика)
  - Marketing в IT-смежной сфере (digital каналы / IT-продукты) → `marketing_manager`
  - Teaching / L&D → `project_manager` / `product_manager` 
  - Юрист / legal → `compliance_officer` / `legal_counsel` (off-index с `marketEvidence`)
  - Sales B2B → `sales_manager` / `customer_success_manager` (off-index)
- **Не рекомендуй** фриланс, контент, консалтинг как основные направления (если кандидат не просил).
- **Хайп-фильтр**: не переоценивай направления с высоким порогом входа (12+ мес) или перегретым рынком.

### Близость перехода (%)

| Диапазон | Значение |
|----------|----------|
| 95-100% | Та же роль, другой работодатель |
| 80-95% | Тот же стек, другая специализация |
| 60-79% | Смежная роль, частичный перенос |
| 40-59% | Серьезный переход |
| < 40% | Фактически новая карьера (не включай без веской причины) |

## Формат ответа
Верни JSON строго по схеме DirectionsOutput. Поле `directions` - массив из 10-14 объектов,
**отсортированный по `score` DESC**. Поля `type` и `bridgeTo` не заполняй (их нет в схеме Phase 1).
